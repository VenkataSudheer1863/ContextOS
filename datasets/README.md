## Overview

**ContextBench** is the official benchmark dataset collection for the ContextOS framework. It provides
evaluation and fine-tuning data across the full context lifecycle: multi-task context management,
retrieval quality, compression fidelity, embedding model fine-tuning, and LLM instruction tuning.

Each subset targets a distinct capability of ContextOS and supplies structured (input, expected_output, metadata)
records that can be used for unit-level benchmarking, regression testing, model fine-tuning, and
research reporting. All subsets are generated reproducibly from fixed random seeds using only Python
standard library code, with no external data dependencies at generation time.

## Subset Descriptions

### 1. `context_bench` â€” Multi-Task Context Management (70,000 examples)

The primary long-horizon context management benchmark. Tests an agent's ability to reason over
collections of 10â€“50 ContextItems across eight task types and six professional domains.
Generated by `generate_context_bench.py` (seed 42).

**Task types:** `summarize_session`, `track_state`, `detect_contradiction`, `recall_procedure`,
`synthesize_facts`, `answer_from_memory`, `identify_gaps`, `timeline_reconstruction`

**Domains:** `software_engineering`, `research`, `customer_service`, `healthcare`, `legal`, `finance`

**Evaluation metrics:** Answer accuracy, ROUGE-L against reference answer, relevant item recall

**Schema:**

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | Unique example identifier (format: `cb-{split}-{index}`) |
| `task` | `string` | Natural-language task instruction |
| `context_items` | `list[object]` | 10â€“50 ContextItem objects (see sub-schema below) |
| `ground_truth` | `string` | Reference answer for the task |
| `relevant_item_ids` | `list[string]` | IDs of context items most relevant to the task |
| `metadata` | `object` | `task_type`, `domain`, `context_length_tokens`, `num_relevant_items`, `difficulty` |

**`context_items[*]` sub-schema:**

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | Item identifier (format: `ctx-{index}`) |
| `content` | `string` | Text content of the context item |
| `type` | `string` | One of `observation`, `decision`, `statement`, `action`, `question`, `fact`, `event`, `note` |
| `timestamp` | `integer` | Unix epoch timestamp |
| `importance` | `float` | Pre-assigned importance score in [0.1, 1.0] |

---

### 2. `retrieval_bench` â€” Retrieval Evaluation (30,000 examples)

Tests the `RetrievalEngine` across dense, sparse, and hybrid retrieval modes. Each example provides
a natural-language query, a corpus of 50â€“100 candidate items with varying relevance levels, and
ground-truth relevant and irrelevant item IDs. Difficulty levels cover obvious matches (easy),
paraphrase matches (medium), and indirect/inferential matches (hard).
Generated by `generate_retrieval_bench.py` (seed 7).

**Domains:** `technology`, `science`, `business`, `medical`

**Evaluation metrics:** Recall@k, MRR (Mean Reciprocal Rank), NDCG@10, precision@5

**Schema:**

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | Unique example identifier (MD5 hash prefix) |
| `query` | `string` | Natural-language search or task query |
| `corpus_items` | `list[object]` | 50â€“100 candidate ContextItem objects |
| `relevant_ids` | `list[string]` | Ground-truth relevant item IDs (1â€“5 per example) |
| `irrelevant_ids` | `list[string]` | Confirmed irrelevant item IDs |
| `metadata` | `object` | `split`, `domain`, `difficulty` (easy / medium / hard), `num_relevant` |

---

### 3. `compression_bench` â€” Compression Fidelity (30,000 examples)

Tests the full `CompressionEngine` strategy suite: extractive (TF-IDF sentence ranking),
abstractive (LLM-based summarisation), hierarchical (three-level: dedup, extraction, keyword),
and progressive (iterative tightening with quality floor). Each example contains a source passage,
a simulated ~40% compression, and pre-computed quality metrics.
Generated by `generate_compression_bench.py` (seed 42).

**Domains:** `technology`, `science`, `business`, `medical`

**Evaluation metrics:** ROUGE-L, Jaccard semantic similarity, information_preserved composite score, compression ratio

**Schema:**

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | Unique example identifier (MD5 hash prefix) |
| `original_text` | `string` | Source passage (200â€“1,000 words) |
| `compressed_text` | `string` | Reference compression at ~40% of original word count |
| `compression_ratio` | `float` | `compressed_words / original_words` |
| `rouge_l_score` | `float` | ROUGE-L F1 between original and compressed text |
| `semantic_similarity` | `float` | Jaccard word-set similarity between original and compressed |
| `information_preserved` | `float` | Composite score: `0.6 * rouge_l + 0.4 * jaccard` |
| `metadata` | `object` | `split`, `domain`, `original_word_count`, `compressed_word_count`, `index` |

---

### 4. `embedding_finetune` â€” Embedding Model Fine-Tuning Data (140,000 examples)

Contrastive triplet data for fine-tuning dense retrieval embedding models
(e.g., BGE-M3, E5-Large-V2) on the ContextOS retrieval task distribution.
Each example contains an anchor query, a positive (relevant) passage, and a hard negative
(superficially similar but irrelevant) passage.
Generated by `generate_embedding_finetune.py`.

**Evaluation metrics:** Triplet loss, MRR on held-out retrieval queries, embedding alignment

**Schema:**

| Field | Type | Description |
|-------|------|-------------|
| `query` | `string` | Search query used as the anchor |
| `positive` | `string` | Passage that is semantically relevant to the query |
| `negatives` | `list[string]` | Hard-negative passages (typically 3): same domain, overlapping vocabulary, different meaning |
| `metadata` | `object` | `domain`, `query_type`, `entity` |

---

### 5. `llm_finetune` â€” LLM Instruction Fine-Tuning Data (30,000 examples)

Instruction-response pairs for supervised fine-tuning of language models on ContextOS
context management tasks. Covers all eight task types from `context_bench` and all six domains,
formatted as multi-turn conversation data compatible with standard chat fine-tuning pipelines.
Generated by `generate_llm_finetune.py`.

**Format:** Alpaca-style `{instruction, input, output}` triples

**Evaluation metrics:** Instruction following accuracy, ROUGE-L on held-out responses, task completion rate

**Schema:**

| Field | Type | Description |
|-------|------|-------------|
| `instruction` | `string` | System-level instruction (e.g., "You are a context-aware agent. Given the following memory context, answer the question.") |
| `input` | `string` | Formatted `<context>` block with numbered memory items followed by the task question |
| `output` | `string` | Reference answer to the task question |
| `metadata` | `object` | `task_type`, `domain`, `context_tokens`, `model` |

## Intended Use Cases

- **Framework evaluation:** Measuring the performance of ContextOS subsystems against fixed baselines
  across software versions.
- **Ablation studies:** Isolating the contribution of individual task types, domains, compression
  strategies, or retrieval modes.
- **Reproducible research:** Publishing results that reference a fixed, versioned dataset so that
  comparisons across papers are fair.
- **Embedding model fine-tuning:** The `embedding_finetune` subset provides contrastive triplets for
  adapting dense retrieval models to the ContextOS task distribution.
- **LLM instruction tuning:** The `llm_finetune` subset provides instruction-response pairs for
  fine-tuning language models on all eight context management task types.
- **Regression testing:** The `retrieval_bench` and `compression_bench` subsets provide fixed
  evaluation sets for monitoring system performance across code changes.

## Limitations and Biases

The following limitations apply to version 1.0.0 of ContextBench.

**Fully synthetic generation.** All examples are programmatically generated using fixed random
seeds and pre-defined sentence pools. The text does not originate from real agent deployments or
genuine long-horizon task histories. Models trained on this data may not generalize to real-world
context distributions without additional fine-tuning on organic data.

**Limited domain vocabulary.** Each domain uses a fixed pool of approximately 25 sentences,
15 entity names, and 10 decision templates. Generated passages may exhibit repetitive surface
patterns that are easier to exploit statistically than real-world text.

**English-only.** All content is English-language. Multilingual contexts, code-mixed inputs, or
non-ASCII content are not represented.

**Approximate token counts.** Token counts in `context_bench` are estimated as
`total_chars / 4` (a character-based heuristic). Models using precise subword tokenizers
(tiktoken, SentencePiece) will observe different values. Token counts in `compression_bench`
are word-based rather than subword-based.

**Compression reference bias.** Reference compressions in `compression_bench` are produced by a
deterministic extractive algorithm that retains the first two sentences then samples randomly from
the remainder. This biases the reference toward earlier sentences in the passage and may favor
extractive over abstractive evaluation metrics.

**Static importance scores.** Importance scores in `context_bench` and `retrieval_bench` are
sampled uniformly from [0.1, 1.0] at generation time and do not model the dynamic re-scoring
that a live ContextOS agent would apply over time.

**Hard-negative quality in `embedding_finetune`.** Hard negatives are sampled from the same
domain corpus rather than adversarially mined, which may underestimate the actual difficulty of
real retrieval tasks. Training on these negatives may yield inflated retrieval metrics.

**Balanced split assumption.** All splits use independent random sampling from the same
generative distribution. Near-duplicate examples may appear across train and test splits due
to the limited vocabulary pool.

## Generation Methodology

ContextBench was constructed entirely programmatically using Python standard library code. Each
generation script is self-contained, reproducible from a fixed seed, and requires no external
data downloads or API calls.

**`generate_context_bench.py` (seed 42)**  
Generates multi-task context management examples. For each example, a domain and task type are
sampled, a set of 10â€“50 ContextItems is synthesized from domain-specific template pools (entities,
actions, decisions, issues), and a ground-truth answer is generated from a matching template.
Relevant items are selected by sorting on importance and sampling from the top candidates.

**`generate_retrieval_bench.py` (seed 7)**  
Generates retrieval evaluation queries paired with corpora of 50â€“100 candidates. Queries are
constructed from domain-specific templates with slot-filling. Relevant items are generated using
the query's key concepts; irrelevant items are drawn from other domains or unrelated templates.
Difficulty is set by the degree of lexical overlap between the query and relevant items.

**`generate_compression_bench.py` (seed 42)**  
Generates source passages of 200â€“1,000 words by concatenating domain sentences with transitional
connectors. A reference compression is produced at ~40% of the original word count by keeping
the first two sentences and greedily sampling the rest. ROUGE-L, Jaccard similarity, and a
composite `information_preserved` score are computed and stored in each record.

**`generate_embedding_finetune.py`**  
Generates contrastive triplets (anchor, positive, negative) for embedding model fine-tuning.
Anchors are task queries; positives are domain-matched passages; negatives are hard-negative
passages from the same domain that match surface-level vocabulary but are topically different.

**`generate_llm_finetune.py`**  
Generates instruction-response pairs in Alpaca/ChatML format. Each example packages a task
instruction, a serialized context window as input, and a reference answer as output. The
distribution mirrors `context_bench` to ensure alignment between evaluation and fine-tuning data.

All generation scripts write flat JSONL files directly to the `datasets/` directory. Files are
UTF-8 encoded with Unix line endings. Each line is a self-contained JSON object.

Run all scripts via the provided entrypoint:

```bash
python datasets/download_datasets.py
```

## Loading the Dataset

### Python (raw JSONL)

```python
import json
from pathlib import Path

DATASETS_DIR = Path("datasets")

def load_subset(name: str, split: str = "test"):
    """Load a ContextBench subset as a list of dicts."""
    path = DATASETS_DIR / f"{name}_{split}.jsonl"
    with path.open(encoding="utf-8") as f:
        return [json.loads(line) for line in f if line.strip()]

context_test      = load_subset("context_bench",      "test")   # 5,000 examples
retrieval_test    = load_subset("retrieval_bench",    "test")   # 2,500 examples
compression_train = load_subset("compression_bench",  "train")  # 25,000 examples
embed_train       = load_subset("embedding_finetune", "train")  # 120,000 examples
llm_train         = load_subset("llm_finetune",       "train")  # 25,000 examples
```

### Hugging Face Datasets

```python
from datasets import load_dataset

# Load a specific subset
ds = load_dataset("contextos/contextbench", "context_bench")
print(ds["test"][0])

# Load all subsets
for name in ["context_bench", "retrieval_bench", "compression_bench",
             "embedding_finetune", "llm_finetune"]:
    ds = load_dataset("contextos/contextbench", name)
    print(f"{name}: {ds}")
```

## Evaluation

Use the `benchmarks/` module to run standardized evaluations against the test splits:

```python
from benchmarks import ContextBenchEvaluator, RetrievalBenchEvaluator

# Evaluate multi-task context management
eval = ContextBenchEvaluator(data_path="datasets/context_bench_test.jsonl")
results = eval.run(agent=my_agent)
print(results.summary())

# Evaluate retrieval
eval = RetrievalBenchEvaluator(data_path="datasets/retrieval_bench_test.jsonl")
results = eval.run(engine=my_retrieval_engine)
print(results.summary())  # Recall@1, Recall@5, MRR, NDCG@10
```
